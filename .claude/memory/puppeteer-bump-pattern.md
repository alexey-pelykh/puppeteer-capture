# Puppeteer Version Bump Pattern

When a new puppeteer version is released, a single commit updates 4 files.

## Commit Convention

- Message: `(imp) puppeteer v{VERSION}`
- Single atomic commit, all changes together
- `npm install` is run to regenerate `package-lock.json` (`npm ci` fails on mismatched lockfile)

## Files Changed (always these 4)

### 1. `package.json`

**devDependencies** — pin both to exact new version (no caret):
```json
"puppeteer": "{NEW_VERSION}",
"puppeteer-core": "{NEW_VERSION}",
```

**peerDependencies** — append new `^major.minor.0` range ONLY if the minor version is new:
- Patch bump (e.g., 24.6.0 -> 24.6.1): NO peerDependencies change (covered by existing `^24.6.0`)
- Minor bump (e.g., 24.5.0 -> 24.6.0): ADD `|| ^24.6.0` to the peerDependencies range

### 2. `.github/workflows/ci.yml`

Two changes:
- **`PUPPETEER_VERSION` env var** (build job): update to new version
- **Integration test matrix** (`puppeteer-version` list): append new version to the list

### 3. `.github/workflows/publish.yml`

- **`PUPPETEER_VERSION` env var**: update to new version

### 4. `package-lock.json`

- Regenerated by `npm install` after the above changes (`npm ci` fails on mismatched lockfile)

## Workflow

1. Create branch `imp/puppeteer-v{VERSION}`
2. Make the 4-file changes + `npm install`
3. Commit: `(imp) puppeteer v{VERSION}`
4. Open PR, merge to main
5. Create GitHub Release (bumps package minor version)
